# План: Этап 6 — Платформа сравнения стратегий

## Контекст

После завершения Этапа 5 (Emergency Mesh) в рабочей области есть:

- `GreedyAllocator` — циклическое распределение по подходящим агентам.
- `AuctionAllocator` — минимизация стоимости по расстоянию, батарее и предпочтительной роли.
- `ConnectivityAwareAllocator` — аукцион + оптимизация доступности сети для размещения ретранслятора.
- 4 типа сценариев: `Coverage`, `DynamicAuction`, `Partition`, `EmergencyMesh`.
- Метрики за прогон (`RunMetrics`) и агрегация (`AggregateMetrics`).
- Детерминированная внутрипроцессная симуляция через `ScenarioRunner::run_with(scenario, config, allocator)`.

Этап 6 превращает проект в исследовательскую платформу, делая сравнение стратегий ключевой функцией. Вместо разовых бинарников, которые сравнивают 2 стратегии на 1 сценарии, платформа будет запускать 1000 сценариев × несколько стратегий × несколько профилей сети/отказов и выдавать структурированные отчёты сравнения.

Долгосрочная цель (согласно DRONE_A.1.md / DRONE_B.1.md) — сравнить:

1. Централизованный планировщик (оптимальный эталон с полным глобальным знанием).
2. Жадный децентрализованный (существующий).
3. Аукционный (существующий).
4. С учётом ретрансляторов / связности (существующий).
5. CBBA (Consensus-Based Bundle Algorithm) — поздний этап.

## Контекст исследования

DRONE_A.1.md и DRONE_B.1.md устанавливают, что проект должен стать исследовательской платформой для сравнения стратегий координации на эталонных миссиях. Текущая кодовая база уже имеет примитивы (аллокаторы, сценарии, метрики, детерминированный модуль запуска). Этап 6 строит *инфраструктуру сравнения* и *дополнительные метрики*, необходимые для строгой оценки.

## Шаги реализации

### Шаг 1: Абстракция стратегии (`swarm-alloc`)

**Файлы:**
- `crates/swarm-alloc/src/strategy.rs` (новый)
- `crates/swarm-alloc/src/lib.rs`

Создать трейт `Strategy`, который оборачивает любой `Allocator` и предоставляет метаданные:

```rust
pub trait Strategy: Allocator {
    fn name(&self) -> &'static str;
    fn description(&self) -> &'static str;
}
```

Реализовать `Strategy` для:
- `GreedyAllocator`
- `AuctionAllocator`
- `ConnectivityAwareAllocator`

Добавить `StrategyRegistry`, который хранит `Vec<Box<dyn Strategy>>` и может итерироваться по зарегистрированным стратегиям.

### Шаг 2: Централизованный планировщик — базовая линия (`swarm-alloc`)

**Файлы:**
- `crates/swarm-alloc/src/centralized.rs` (новый)
- `crates/swarm-alloc/src/lib.rs`

Реализовать `CentralizedPlanner` — аллокатор с *оракуловским* доступом к полному состоянию сценария (все позы агентов, все задачи, без ограничений связи). Решает задачу двудольного сопоставления с помощью венгерского алгоритма (или жадного, если `petgraph` недоступен) для минимизации суммарной стоимости (расстояние + штраф за батарею + предпочтение роли).

Это **базовая линия верхней границы**: ни одна децентрализованная стратегия не должна превзойти его в идеальных условиях. Отвечает на вопрос «сколько мы теряем из-за децентрализации?»

`CentralizedPlanner` реализует `Allocator`, но требует ссылку на `Scenario` при конструировании. Для инфраструктуры сравнения он будет создаваться на каждый прогон с текущим сценарием.

### Шаг 3: Новые метрики (`swarm-metrics`)

**Файлы:**
- `crates/swarm-metrics/src/metrics.rs`

Расширить `RunMetrics` и `AggregateMetrics`:

```rust
// Покрытие: доля площади, покрытой назначенными агентами (0.0..=1.0)
pub coverage_progress: f64,
// Отправлено байт: суммарный размер payload всех сообщений
pub bytes_sent: u64,
// Возраст stale-состояния: макс. разница между локальным tick и последним известным удалённым tick
pub stale_state_age_ticks: u64,
// Запас батареи (если включена модель батареи)
pub battery_margin_min: f64,
pub battery_margin_avg: f64,
```

Обновить `AggregateMetrics::from_runs` для усреднения новых полей.

Обновить `Display for AggregateMetrics` для включения новых полей.

### Шаг 4: Профили сети и отказов (`swarm-scenarios`)

**Файлы:**
- `crates/swarm-scenarios/src/profiles.rs` (новый)
- `crates/swarm-scenarios/src/lib.rs`

Определить переиспользуемые структуры `NetworkProfile` и `FailureProfile`:

```rust
pub struct NetworkProfile {
    pub name: &'static str,
    pub packet_loss_rate: f64,
    pub latency_ticks: u64,
    pub latency_per_hop: u64,
}

pub struct FailureProfile {
    pub name: &'static str,
    pub failure_count: usize,
    pub failure_tick_range: (u64, u64),
}
```

Предоставить модуль `StandardProfiles` с предопределёнными профилями:

- Сети: `Ideal`, `LightLoss`, `MediumLoss`, `HeavyLoss`, `HighLatency`, `PartitionProne`
- Отказы: `NoFailures`, `SingleFailure`, `MultipleFailures`, `CascadeFailure`

### Шаг 5: Оснастка сравнения (`swarm-sim`)

**Файлы:**
- `crates/swarm-sim/src/benchmark.rs` (новый)
- `crates/swarm-sim/src/lib.rs`

Создать `BenchmarkHarness`, который запускает:

```
for each seed in 0..1000:
  for each strategy in strategies:
    for each network_profile in network_profiles:
      for each failure_profile in failure_profiles:
        build scenario with seed + profile params
        run ScenarioRunner::run_with(...)
        collect RunMetrics
```

Инфраструктура формирует `ComparisonReport`:

```rust
pub struct ComparisonReport {
    pub strategy_names: Vec<String>,
    pub profile_names: Vec<String>,
    pub results: HashMap<(String, String), AggregateMetrics>,
}
```

Реализовать `Display for ComparisonReport`, который печатает markdown-совместимую таблицу:

```
| Стратегия | Профиль | Успех | Обнаружение | Перераспределение | Покрытие | Сообщения | Доступность |
|-----------|---------|-------|-------------|-------------------|----------|-----------|-------------|
```

### Шаг 6: Бинарник сравнения стратегий (`swarm-examples`)

**Файлы:**
- `crates/swarm-examples/src/bin/strategy_comparison.rs` (новый)

Главный бинарник сравнительных прогонов:

1. Зарегистрировать все стратегии: `Greedy`, `Auction`, `ConnectivityAware`, `Centralized`.
2. Зарегистрировать все профили из `StandardProfiles`.
3. Запустить оснастку сравнительных прогонов.
4. Напечатать отчёт сравнения.
5. Проверить инварианты (например, `Centralized` коэффициент успеха >= `Greedy` коэффициент успеха при идеальной сети).
6. Код выхода `0` при успехе, `1` при нарушении инварианта.

### Шаг 7: Обновление существующих сценариев для новых метрик

**Файлы:**
- `crates/swarm-sim/src/runner.rs`

Обновить `ScenarioRunner::run_with` для вычисления и заполнения:
- `coverage_progress` — геометрическое покрытие поз задач назначенными агентами по сравнению с общей площадью.
- `bytes_sent` — сумма `msg.payload.len()` по всем сообщениям.
- `stale_state_age_ticks` — макс. `current_tick - entry.last_heartbeat_tick` по всем живым агентам.
- `battery_margin_min` / `battery_margin_avg` — мин/средняя батарея живых агентов в конце прогона.

## Стратегия тестирования

### Категория 1: Чистые unit-тесты (без рефакторинга)

- `strategy.rs`: добавление/итерация в `StrategyRegistry`.
- `centralized.rs`: `CentralizedPlanner` назначает все задачи при достаточном числе агентов; возвращает пустой результат при отсутствии агентов.
- `profiles.rs`: `StandardProfiles` содержит ожидаемые имена профилей и диапазоны параметров.
- `benchmark.rs`: агрегация `ComparisonReport` и форматирование вывода.

### Категория 2: Лёгкие интеграционные тесты (внутрипроцессная симуляция)

- Тест бинарника `strategy_comparison.rs`: запустить маленький сравнительный прогон (10 seeds × 2 стратегии × 2 профиля) и проверить структуру отчёта и проверки инвариантов.
- Проверить, что `CentralizedPlanner` превосходит или совпадает с `GreedyAllocator` при идеальной сети (детерминированно, без потерь пакетов, без разделения сети).
- Проверить, что `ConnectivityAwareAllocator` достигает более высокой `network_availability`, чем `AuctionAllocator`, на профиле `PartitionProne` в сценарии Emergency Mesh.

### Категория 3: Тяжёлые end-to-end тесты (полный benchmark)

- Запустить `strategy_comparison` с 1000 seeds, всеми 4 стратегиями, всеми стандартными профилями.
- Проверить, что общее время выполнения разумно (< 5 минут на текущем железе).
- Проверить, что отчёт содержит все ожидаемые строки и нет значений NaN.
- Ручной просмотр отчёта на аномалии (например, отрицательный коэффициент успеха, невозможное покрытие > 1.0).

## Риски и компромиссы

| Риск | Влияние | Митигация |
|------|---------|-----------|
| CentralizedPlanner требует знания на уровне сценария, нарушая абстракцию `Allocator`. | Среднее | Реализовать как особый случай `Allocator`, который предвычисляет оптимальные назначения из данных сценария при конструировании. Документировать как стратегию только для измерений. |
| Метрика прогресса покрытия дорого вычислять геометрически. | Низкое | Использовать грубую аппроксимацию сеткой (например, 20×20) или прокси на основе задач (доля задач с агентами в пределах sensor range). |
| Время выполнения сравнительных прогонов растёт как O(seeds × strategies × networks × failures). | Среднее | По умолчанию использовать сокращённую матрицу (например, 100 seeds) в CI/тестах; полные 1000 seeds только в режиме release/измерений. |
| Добавление новых метрик ломает обратную совместимость сохранённых JSON `RunMetrics`. | Низкое | Добавить `#[serde(default)]` на новые поля. Существующие данные для повторного прогона без новых полей десериализуются безопасно. |
| Откладывание CBBA создаёт пробел в сравнении стратегий. | Низкое | Документировать CBBA как «запланировано для v0.7». Фреймворк сравнения поддерживает добавление новых стратегий без изменений кода. |

## Открытые вопросы

1. **Точность метрики покрытия**: покрытие должно быть на основе задач (назначенные агенты рядом с позами задач) или на основе площади (дискретизация сеткой)? Подход на основе задач дешевле, но менее содержателен для сценариев поиска и спасения.
2. **Глубина модели батареи**: в v0.3+ батарея статическая. Следует ли в Этапе 6 добавить простую модель разряда (`battery -= distance * energy_per_meter`) или оставить батарею статической и использовать `battery_margin` как прокси качества назначения?
3. **Объём CBBA**: следует ли сделать CBBA быстрым выигрышем в Этапе 6 (упрощённое bundle-building без полного консенсуса) или отложить в отдельный этап с правильными расширениями протокола сообщений?
4. **Формат отчётов**: Markdown-таблица читается человеком, но плохо парсится программно. Следует ли также выдавать JSON/CSV для последующего анализа (Python/Polars)?
5. **Статистическая значимость**: текущие агрегированные метрики используют простые средние. Следует ли добавлять доверительные интервалы или использовать Mann-Whitney U-test для сравнения стратегий? Это можно отложить на поздний исследовательский этап.

## Что могло сломаться

- **Поведение**: Существующие бинарники сценариев (`coverage_with_failure`, `dynamic_auction` и др.) не должны менять поведение. Новые поля метрик имеют значения по умолчанию serde, поэтому существующие пути кода не затронуты.
- **API/контракты**: Трейт `Allocator` не меняется. `Strategy` — новый super-trait. Существующие аллокаторы продолжают работать.
- **Производительность**: `ScenarioRunner` получает дополнительное вычисление метрик на каждый tick. Расчёт coverage-сетки может добавить ~5-10% накладных расходов. Митигация: вычислять coverage только когда используется поле `coverage_progress` (feature flag или config flag).
- **Детерминизм**: Новые метрики (`bytes_sent`, `stale_state_age`) должны вычисляться детерминированно из того же seed и последовательности tick'ов. Никакой новой случайности вводить нельзя.
- **Интеграция**: `CentralizedPlanner` — только для измерительных прогонов и не должен использоваться в multi-process или реальных полётных сценариях. Документировать это явно.
