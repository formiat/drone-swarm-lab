# Сравнение DRONE_A.6 и DRONE_B.6. Итоговый набор направлений

> Сессия: май 2026. Синтез DRONE_A.6 и DRONE_B.6.

---

## Насколько отличаются документы

**Структура:** DRONE_B.6 — три высокоуровневых несовместимых направления. DRONE_A.6 —
семь конкретных вариантов плюс выявленный ближайший общий шаг.

**Главное что есть в A.6 и нет в B.6 — M11 hardening.** A.6 обнаружил конкретные
баги в текущей реализации: поля `mission` и `scenario` в JSON/CSV пустые,
`benchmark_run_id` в `--mission all` остаётся coverage-specific, README-числа из
10-seed quick run, нет proptest для distributed CBBA. B.6 считает M11 чистым — это
неверно.

**Mission DSL:** в B.6 это один из пунктов направления "Research Platform". A.6
выделяет его в отдельный общий инфраструктурный шаг, который нужен почти для всех
веток — SAR v2, Infrastructure Inspection, regression CI. Это точнее.

**Реальное железо:** оба документа сходятся что преждевременно. A.6 формулирует
почему конкретнее: без DSL, safety layer и стабильного report API это будет дорогой
демонстрацией без исследовательской глубины.

**Visualization и CBBA Robustness:** A.6 выделяет их явно, B.6 не упоминает совсем.

**Где B.6 лучше:** высокоуровневый фрейм — три несовместимых пути — более чистая
ментальная модель для принятия решения о цели. A.6 даёт 7 вариантов без явного
"выбери цель сначала".

## Итоговый набор направлений

Структура из двух слоёв: общий ствол и ветки.

### Общий ствол (нужен в любом случае, не требует выбора)

**1. M11 hardening**

Исправить конкретные баги в текущем report:

- заполнить `mission` и `scenario` в JSON/CSV (сейчас пустые);
- починить `benchmark_run_id` для `--mission all` (сейчас coverage-specific);
- добавить proptest для distributed CBBA на случайных топологиях;
- запустить полноценный 1000-seed прогон и обновить README-числа.

Это не новая исследовательская ветка — доведение текущего результата до состояния,
на которое можно опираться дальше.

**2. Mission DSL**

Декларативное описание сценариев (YAML / RON / JSON). Нужен как инфраструктура для
всех веток. Текущие три миссии (Coverage, EmergencyMesh, SAR) уже показали
минимальный нужный набор полей — риск зацементировать плохую модель минимален.

Что даёт:
- сценарии становятся воспроизводимыми артефактами, а не только кодом;
- новые benchmark cases добавляются без перекомпиляции;
- CI может гонять набор конфигов как test suite;
- основа для всех последующих миссий и веток.

---

### Ветки (выбрать одну по цели проекта)

**Ветка A — Исследовательская глубина**

Что: SAR v2 (confidence map, repeated scans, false positives, target belief) +
CBBA Robustness (scaling, packet loss stress, convergence distributions,
retransmission policy).

Зачем: publishable result с методологически сильным сравнением алгоритмов. Стратегии
начнут отличаться не только allocation quality, но и поведением под неопределённостью
и при деградации сети.

Где пригодится: swarm robotics research, distributed systems research, статьи по
устойчивости алгоритмов.

**Ветка B — Прикладная демонстрация**

Что: Infrastructure Inspection (новая reference mission — линии, трубопроводы,
solar farm) + Visualization / Replay UI.

Зачем: понятный industrial use case + возможность показать поведение алгоритмов
глазами, а не только числами.

Где пригодится: power line / pipeline / perimeter inspection, presentation demos,
debugging, teaching.

**Ветка C — Путь к реальному железу**

Что: Safety Layer сначала (geofence, separation, collision avoidance-lite),
PX4 / MAVLink / SITL потом.

Зачем: перенести алгоритмы координации на реальные дроны. Архитектура
`AgentNode<T: Transport>` это допускает — транспорт подменяем.

Где пригодится: реальные беспилотные системы, SITL demos, hardware-in-the-loop.

Ограничение: без Safety Layer этот путь преждевременен. Без DSL и стабильного report
API интеграция будет дорогой демонстрацией без исследовательской глубины.

---

## Совместимость веток

Ветки не полностью несовместимы: Safety Layer полезен для A и B, Visualization
полезна везде. Но как основное направление нужно выбрать одну — иначе работа
размажется без фокуса.

---

## Ключевой вопрос

Прежде чем брать M12, имеет смысл ответить:

> Цель — publishable research result, понятная industrial демонстрация,
> или движение к реальным дронам?

Ответ определяет, что строить после общего ствола.
