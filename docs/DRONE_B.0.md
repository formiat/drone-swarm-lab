
### Полный стек любой системы

```
┌──────────────────────────────────────────────────────┐
│  Mission layer        миссия, цели, приоритеты       │
├──────────────────────────────────────────────────────┤
│  Coordination layer   task allocation, consensus,    │
│                       formation, роли                │
├──────────────────────────────────────────────────────┤
│  Communication layer  протоколы, mesh, routing,      │
│                       reliability, security          │
├──────────────────────────────────────────────────────┤
│  Navigation layer     планирование пути, CA,         │
│                       локализация (GPS / SLAM)       │
├──────────────────────────────────────────────────────┤
│  Control layer        PID, траектория, стабилизация  │
├──────────────────────────────────────────────────────┤
│  Hardware layer       FC, моторы, сенсоры            │
└──────────────────────────────────────────────────────┘
```

Большинство академических работ и open-source проектов берут **один слой** и решают его в изоляции.
Это главная структурная проблема поля.

## 2. Карта существующих решений и дыры

| Слой | Что есть | Чего нет |
|---|---|---|
| Hardware | PX4, ArduPilot — зрелые | — |
| Control | PX4 SITL, Gazebo | — |
| Navigation | SLAM, VIO — библиотеки есть | Интеграция в swarm-контекст слабая |
| Communication | MAVLink, zenoh, DDS | **Нет серьёзной модели ненадёжной сети** |
| Coordination | CBBA-статьи, игрушечные impl | **Нет production-качества реализации** |
| Mission | QGroundControl, Mission Planner | Только для флота, не для роя |
| **Полный стек** | Skybrush (шоу), Aerostack2 | **Нет общей открытой платформы** |
| **Тестирование** | Gazebo (оптимистичная сеть) | **Нет adversarial/failure-testing среды** |
| **Гетерогенные роевые системы** | Статьи | **Реализаций почти нет** |
| **Fleet open source** | AP_Cloud (рудиментарно) | **Коммерческий рынок полностью закрытый** |

**Симуляторы существуют** (Gazebo, AirSim, ARGoS, VMAS) — делать ещё один нет смысла.
**Control layer существует** (PX4, ArduPilot) — трогать незачем.

## 3. Выбранное направление: Swarm Coordination Runtime

### Что это

Библиотека/демон на Rust, которая запускается на каждом агенте (физическом или симулируемом)
и реализует **coordination layer** — слой между "железо летит" и "миссия выполняется".

```
┌─────────────────────────────────────────────────────┐
│              Mission / Application                  │
├─────────────────────────────────────────────────────┤
│         Swarm Coordination Runtime  ← ПРОЕКТ        │
│  • membership + heartbeat (кто жив)                 │
│  • distributed task allocation (CBBA / auction)     │
│  • failure detection + reallocation                 │
│  • shared state consensus                           │
│  • communication model (partial connectivity)       │
├─────────────────────────────────────────────────────┤
│    Transport (pluggable: UDP / MAVLink / zenoh)      │
├─────────────────────────────────────────────────────┤
│              PX4 / ArduPilot / симулятор             │
└─────────────────────────────────────────────────────┘
```

### Почему это закрывает реальный gap

Любой кто строит swarm-систему упирается в этот слой и пишет его заново с нуля.
Open-source runtime = инфраструктура на которой строят другие.

Аналог: что Erlang/OTP делает для телекомов — но для роёв дронов.

### Инкрементальный roadmap

- **v0.1** — heartbeat + membership + один сценарий (coverage, 1 агент падает → перераспределение)
- **v0.2** — CBBA, динамические задачи
- **v0.3** — pluggable transport (UDP / MAVLink / zenoh)
- **v0.4** — partial connectivity, mesh routing
- **v0.5** — Byzantine fault tolerance

Каждая версия уже полезна сама по себе. Не нужно делать всё чтобы было ценно.

## 4. Как разрабатывать без физических дронов

### Ключевое понимание

**Coordination runtime — это задача distributed systems, а не задача дронов.**

"Дрон" для кода — это агент с ID, позицией и inbox/outbox.
Неважно откуда берётся позиция: реальный IMU, Gazebo, или `position += velocity * dt` в тесте.

Никто не разрабатывает Raft, Paxos или TCP на реальном железе.
Ты разрабатываешь протокол — и тестируешь протокол.

### Пирамида тестирования

```
                    ▲
                   ╱ ╲
                  ╱ 4 ╲   PX4 SITL + Gazebo
                 ╱─────╲  тяжело, редко, для финальной интеграции
                ╱   3   ╲
               ╱─────────╲ N процессов + fake physics
              ╱           ╲ UDP loopback, средне
             ╱      2      ╲
            ╱───────────────╲ In-process async + simulated network
           ╱                 ╲ быстро, большинство тестов здесь
          ╱         1         ╲
         ╱─────────────────────╲ Pure unit tests
        ╱  алгоритм без физики  ╲ мгновенно
       ▼─────────────────────────▼
```

---

### Уровень 1 — Чистые unit-тесты алгоритма

Тестируешь логику координации напрямую. Никакой физики, никаких дронов.
Позиция агента — просто `(f32, f32)`.

```rust
#[test]
fn test_cbba_reallocates_on_agent_death() {
    let mut swarm = SwarmSim::new(5);
    swarm.assign_tasks(vec![Task::cover(zone_a), Task::cover(zone_b)]);
    swarm.kill_agent(2);
    swarm.run_coordination_rounds(10);
    assert!(swarm.all_tasks_assigned());
}
```

---

### Уровень 2 — In-process симуляция с управляемой сетью

N агентов как N tokio-задач в одном процессе.
Между ними — канал с настраиваемой деградацией.

```rust
struct SimNetwork {
    packet_loss: f32,                       // 0.3 = 30% потерь
    latency_ms: u64,                        // задержка
    partition: Vec<(AgentId, AgentId)>,     // разорванные пары
}
```

Возможности:
- Тестировать сходимость при 30% потерях пакетов
- Симулировать сетевые разделения (split-brain)
- Запускать 1000 сценариев в секунду (всё в памяти)
- Property-based тестирование: `proptest` генерирует случайные паттерны отказов,
  проверяешь инварианты ("все задачи всегда назначены если жив хотя бы 1 агент")

**Это основная тестовая среда.** Именно так тестируют etcd, CockroachDB, TiKV.

---

### Уровень 3 — N процессов + fake physics

Каждый агент — отдельный OS-процесс. Связь через UDP loopback `127.0.0.x`.
Позиция обновляется простым интегратором:

```
position += velocity * dt
```

Проверяешь что протокол работает через реальный сетевой стек.
`kill -9` на процесс = идеальный тест failure detection.

---

### Уровень 4 — PX4 SITL

PX4 запускает N экземпляров на одном ПК, каждый симулирует дрон с реальной физикой полёта.
Coordination runtime подключается через MAVLink.

```bash
Tools/simulation/sitl_multiple_run.sh -n 5
```

Тяжело (~500 MB RAM на экземпляр). Используется редко — финальная проверка интеграции.

---

### Что физически нужно

| Что | Зачем |
|---|---|
| Ноутбук с Rust | Уровни 1, 2, 3 — всё здесь |
| 8+ GB RAM | PX4 SITL для уровня 4 |
| Ubuntu / WSL2 | PX4 SITL нативно под Linux |
| **Физические дроны** | **Не нужны** |

---

### Аналогия

Разработчики etcd тестируют Raft-консенсус на тысячах симулированных сценариев
с потерями пакетов и падениями узлов — всё в памяти, на одном сервере,
без единого "настоящего" кластера в процессе разработки.

Coordination runtime для роя дронов устроен так же.
Физические дроны — это один из возможных транспортов, подключаемый в самом конце.
